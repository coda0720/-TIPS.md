```text
GitHub

＋
```

最初にログイン・購入者登録・権限管理などの仕組みを作れば、その後は購入者ごとに手作業する必要はありません。

```text
最初に仕組みを作る

↓

```

# 使用するサービス

| サービス               | 役割                |          固定費 |
| ------------------ | ----------------- | -----------: |
| GitHub             | サイトのファイルやプログラムを保存 |           0円 |
| Cloudflare Pages   | サイトをインターネット上に公開   |           0円 |
| Cloudflare Workers | 決済確認・ログイン・権限判定を実行 |        0円枠あり |
| Cloudflare D1      | 購入者・購入履歴・プロンプトを保存 |        0円枠あり |
| PayPal             | 3,000円の決済を受け付ける   | 月額0円・決済手数料あり |

---

# それぞれの役割

## GitHub

サイトの元データを保存します。

```text
・HTML
・CSS
・JavaScript
・Cloudflare Workersのプログラム
```

販売するプロンプト本文は、公開GitHubに直接保存しない方が安全です。

---

## Cloudflare Pages

購入ページ、ログイン画面、プロンプト一覧画面などを公開します。

```text
販売ページ
ログインページ
購入者専用ページ
```

---

## Cloudflare Workers

サイトの裏側で、自動処理を行います。

```text
・PayPalの支払い確認
・購入者の登録
・ログイン確認
・閲覧権限の判定
・返金時の利用停止
・プロンプトデータの取得
```

---

## Cloudflare D1

購入者情報などを保存するデータベースです。

主に以下を保存します。

```text
・購入者情報
・購入履歴
・ログイン情報
・閲覧権限
・プロンプト本文
```
### 7,000件は一括登録できる

現在のプロンプトが次のどれかで保存されていれば、一括変換できます。

- HTML内に並んでいる
- JavaScriptの配列になっている
- JSONファイルになっている
- CSVやExcelで管理している
- Markdownファイルに並んでいる

## 処理イメージ

1. 現在の7,000件を読み取る
2. JSONまたはCSVに変換
3. 一括登録用プログラムを1回実行
4. D1へ7,000件まとめて登録

---

## PayPal

3,000円の決済を受け付けます。

支払いが完了すると、PayPalからCloudflare Workersへ自動通知を送れます。

この通知をWebhookと呼びます。

---

# 完成後の自動処理

```text
1．購入者がPayPalで支払う
        ↓
2．PayPalが支払い完了を通知する
        ↓
3．Cloudflare Workersが通知を受け取る
        ↓
4．本当にPayPalから届いた通知か確認する
        ↓
5．金額が3,000円か確認する
        ↓
6．購入者情報をD1に保存する
        ↓
7．閲覧権限を有効にする
        ↓
8．購入者がログインする
        ↓
9．購入済みか自動判定する
        ↓
10．プロンプト一覧を表示する
```

---

# 最初に作る必要があるもの

## 1．購入者情報

D1に購入者を保存する表を作ります。

```text
users

・ユーザー番号
・メールアドレス
・パスワード情報
・閲覧権限
・登録日時
・利用停止状態
```

パスワードは、そのまま保存せず、暗号化した状態で保存します。

---

## 2．購入履歴

```text
purchases

・PayPal取引番号
・購入者メール
・購入金額
・通貨
・支払い状態
・購入日時
・返金状態
```

PayPal取引番号は、同じ番号を重複登録できないようにします。

これにより、同じ通知が複数回来ても二重登録を防げます。

---

## 3．ログイン状態

```text
sessions

・ユーザー番号
・ログイン用トークン
・有効期限
・作成日時
```

ログインに成功した購入者には、安全なCookieを発行します。

Cookieによって、毎回パスワードを入力しなくてもログイン状態を維持できます。

---

## 4．プロンプト情報

```text
prompts

・プロンプト名
・プロンプト本文
・カテゴリー
・タグ
・登録日時
・公開状態
```

販売するプロンプト本文は、公開HTMLや公開GitHubに直接書かないようにします。

### 避ける構成

```text
公開GitHub
↓
プロンプト全文を保存
↓
購入していない人にも見られる可能性がある
```

### 推奨構成

```text
購入者がログイン
↓
Workersが購入権限を確認
↓
D1からプロンプトを取得
↓
購入者専用画面に表示
```

---

# Workersに作る主な処理

```text
/api/create-order
→ PayPalの注文を作る

/api/capture-order
→ PayPalの支払いを確定する

/api/paypal-webhook
→ PayPalから支払い・返金通知を受け取る

/api/register
→ 購入済みの人を登録する

/api/login
→ メールアドレスとパスワードを確認する

/api/logout
→ ログアウトする

/api/prompts
→ 購入者だけにプロンプトを返す
```

これらを最初に作れば、購入が発生するたびに自動で実行されます。

# 「自分で組み立てる」の意味

「毎回手作業する」という意味ではありません。

```text
最初にプログラムを書く
↓
PayPal・Workers・D1を接続する
↓
動作確認をする
↓
以後はプログラムが自動処理する
```

券売機を最初に作れば、その後は人が毎回切符を手渡さなくてよいのと同じイメージです。

---

# 自動化されないもの

仕組みを完成させても、以下は運営者の作業として残ります。

```text
・新しいプロンプトの追加
・プロンプト本文の修正
・問い合わせ対応
・返金するかどうかの判断
・PayPalの異議申し立て対応
・不正利用への対応
・システムエラーの確認
・PayPalやCloudflareの仕様変更への対応
```

購入者一人ひとりを手動登録する必要はなくなりますが、サイト管理そのものが完全になくなるわけではありません。

# メールを使わない無料構成

完全に固定費0円を優先する場合は、メール送信を使わない方法があります。

```text
1．購入者がPayPalで支払う
        ↓
2．支払い完了ページへ戻る
        ↓
3．購入者がパスワードを設定する
        ↓
4．メールアドレスとパスワードでログインする
```

## メリット

```text
・外部メールサービスが不要
・固定費0円で始めやすい
・購入者登録を自動化できる
```

## デメリット

```text
・パスワード再設定メールを送れない
・パスワードを忘れた人への対応が手動になる
・購入完了メールを自動送信できない
```

---

# メールまで自動化する構成

ログイン案内やパスワード再設定も自動化する場合は、無料枠のあるメールサービスを追加します。

```text
GitHub
＋
Cloudflare Pages
＋
Cloudflare Workers
＋
Cloudflare D1
＋
PayPal
＋
外部メールサービス
```

メールサービスを追加すると、以下も自動化できます。

```text
・購入完了メール
・利用開始メール
・ログイン用リンク
・メールアドレス確認
・パスワード再設定
```

# Supabaseあり・なしの違い

| 項目       | Supabaseあり | Cloudflareだけ        |
| -------- | ---------- | ------------------- |
| データベース   | 用意されている    | D1を使う               |
| 会員登録     | 機能が用意されている | 自分で作る               |
| ログイン     | 機能が用意されている | 自分で作る               |
| セッション管理  | 用意されている    | 自分で作る               |
| 閲覧権限     | 設定中心       | Workersで作る          |
| パスワード再設定 | 用意されている    | 自分で作る               |
| PayPal連携 | 接続処理が必要    | 接続処理が必要             |
| 完成後の自動化  | 可能         | 可能                  |
| 初期設定の難易度 | 比較的簡単      | やや難しい               |
| 使用サービス数  | 増える        | Cloudflare中心にまとめられる |

---

# Cloudflareだけで作るメリット

```text
・固定費0円で始めやすい
・Supabaseを管理しなくてよい
・サイトとデータベースをCloudflareにまとめられる
・購入者が増えても手動登録が不要
・文章中心のサイトなら無料枠で運営しやすい
```

---

# Cloudflareだけで作るデメリット

```text
・最初に作るプログラムが多い
・ログイン機能を自分で作る必要がある
・パスワードの安全管理が必要
・セキュリティ設定を間違える可能性がある
・メール送信には別サービスが必要になりやすい
・エラーが起きた場合は自分で修正する必要がある
```

---

# 必須の安全対策

```text
・PayPal通知が本物か確認する
・支払い金額が3,000円か確認する
・通貨が日本円か確認する
・支払い状態が完了か確認する
・同じPayPal取引番号を重複登録しない
・パスワードをそのまま保存しない
・秘密鍵をGitHubやHTMLに書かない
・安全なCookieを使用する
・未購入者にはプロンプト本文を返さない
・返金時に閲覧権限を停止する
```

支払い完了ページへ移動しただけで、閲覧権限を与えてはいけません。

必ずPayPalから届くWebhookを確認してから権限を付与します。

# 最終判断

Supabaseなしでも、最初に以下を正しく作れば、その後は自動運営できます。

```text
・PayPal決済確認
・購入者登録
・購入履歴保存
・ログイン
・ログイン状態の維持
・閲覧権限判定
・返金時の利用停止
```

ただし、Supabaseを使用する場合よりも、最初に作るプログラムと安全対策が増えます。

```text
初期構築の簡単さ
→ Supabaseが有利

サービス数の少なさ
→ Cloudflareだけの構成が有利

完成後の自動化
→ どちらも可能
```

**固定費0円・サービス数を減らしたい場合は、Cloudflare Pages・Workers・D1・PayPalの構成が適しています。**
